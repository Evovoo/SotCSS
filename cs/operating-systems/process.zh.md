---
title: 进程与线程
---

## 1. What
进程和线程是操作系统最核心的执行抽象。进程是拥有独立资源和虚拟地址空间的执行单位，也是保护边界；线程则是进程内部可被调度的执行路径，共享进程的大部分资源和内存空间。

这个主题还包括进程/线程的状态转换，以及并发执行中必须面对的同步与互斥机制。

## 2. Why
这些抽象之所以重要，是因为现代系统必须同时处理很多任务：

- 进程提供故障隔离和资源隔离
- 线程让单个应用内部能够并行或并发执行
- 状态模型解释了阻塞、唤醒和挂起
- 同步机制用来防止竞态、丢失更新和共享状态不一致

如果不理解这些概念，就无法真正理解系统的响应性、吞吐和并发正确性。

## 3. How
典型的进程状态包括：

- new
- ready
- running
- blocked 或 waiting
- terminated

常见状态转换可以写成：

```text
new -> ready
ready -> running
running -> blocked
blocked -> ready
running -> terminated
running -> ready
```

最后一种转换通常对应抢占发生。

同一进程内的线程通常共享：

- 代码段
- 堆
- 打开的文件
- 地址空间

但每个线程各自拥有：

- 程序计数器
- 寄存器上下文
- 栈

互斥通常通过锁、信号量、管程或原子指令实现。

```text
lock(m)
critical section
unlock(m)
```

基本原则是：任何被多个线程访问的可变共享状态，都必须通过同步保护，或者通过设计消除共享。

## 4. Better
进程的隔离性强于线程，但创建与切换通常更重。线程更轻量，也更适合共享内存协作，但共享本身会带来竞态和可见性问题。

不同同步机制之间也有明显取舍：

- mutex 适合独占访问
- semaphore 适合计数资源和线程间通知
- condition variable 适合等待状态变化
- 无锁技术能减少阻塞，但理解和实现成本更高

还需要区分“为了组织程序结构而并发”和“为了利用多核而并行”。线程可以同时服务这两个目标，但正确性始终依赖明确的协调规则。

## 5. Beyond
真实系统里还必须继续处理：

- 死锁和饥饿
- 优先级反转
- 硬件乱序下的内存可见性
- 用户级线程与内核级线程的差异

更深层的结论是：并发不是简单的“同时做很多事”，而是调度器、内存模型和同步纪律共同定义出来的一种执行契约。
