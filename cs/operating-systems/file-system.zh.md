---
title: 文件系统
---

## 1. What
文件系统负责组织数据在存储设备上的命名、存放、定位、保护和恢复方式。它在底层物理块和持久化机制之上，向上提供“文件”和“目录”这两个核心抽象。

教材中的核心内容包括文件结构、目录组织、分配方式、元数据以及基于 inode 的索引机制。

## 2. Why
文件系统重要，是因为持久化数据不能随着进程结束或机器重启而消失：

- 应用需要稳定的命名与查找能力
- 存储空间必须能被分配和回收
- 元数据要描述权限、归属和布局
- 崩溃后不能让系统长期处于不一致状态

文件系统本质上是逻辑数据组织和物理存储布局之间的桥梁。

## 3. How
一个文件通常由以下部分组成：

- 数据块
- 元数据
- 目录项，负责把名字映射到文件对象

在基于 inode 的系统里，inode 会保存如下元数据：

- 文件类型
- 权限
- 所有者
- 大小
- 时间戳
- 数据块指针

目录项通常只保存“文件名 -> inode 编号”的映射，而不是把全部元数据直接塞进目录项。

常见分配模型：

| Method | Idea | Trade-off |
| --- | --- | --- |
| contiguous | 连续放置块 | 顺序访问快，但扩展不灵活 |
| linked | 每个块指向下一块 | 增长简单，但随机访问差 |
| indexed | 通过索引块或 inode 指向数据块 | 灵活，但元数据开销更高 |

类 Unix 查找路径的大致过程：

```text
path -> directory entries
directory entry -> inode number
inode -> block pointers
block pointers -> file data blocks
```

## 4. Better
inode 模型优于把所有元数据直接放进目录项，因为多个名字可以同时指向同一个底层文件对象，元数据管理也更集中。

索引分配通常比纯链式分配更适合随机访问；连续分配对顺序访问非常高效，但一旦文件动态增长，就容易受碎片和移动成本影响。

这里的核心张力在于性能与灵活性。连续布局提升吞吐，但动态更新、可恢复性和一致性又要求更多间接层和元数据。

## 5. Beyond
现代文件系统还要进一步处理：

- journaling 或 copy-on-write 一致性
- 缓存与回写策略
- extent 分配
- 面向 SSD 的布局优化
- 分布式与网络文件系统

更深层的结论是：文件系统不只是一个目录树，而是一套在性能、耐久性和故障恢复约束下运行的分配与一致性引擎。
