---
title: 传输层
---

## 1. What
传输层负责不同主机上不同进程之间的端到端通信服务。它要在网络层“尽力而为”的分组传输之上，向上提供可靠性、有序性、多路复用和流量控制等进程级语义。

这一层最核心的教材内容，就是 TCP 与 UDP 的对比，以及 TCP 的连接管理和拥塞控制。

## 2. Why
传输层重要，是因为应用并不是以“主机”身份通信，而是以“具有具体服务需求的端点”身份通信：

- 有些应用必须要可靠且有序
- 有些应用更看重低延迟，能容忍丢包
- 一台主机上通常会同时跑很多网络会话
- 发送速率既要考虑接收端能力，也要考虑网络本身状态

这一层是端到端策略被真正显式化的地方。

## 3. How
TCP 提供：

- 面向连接的通信
- 可靠、按序的字节流
- 重传与确认
- 流量控制
- 拥塞控制

TCP 建立连接通常使用三次握手：

```text
client -> SYN
server -> SYN + ACK
client -> ACK
```

连接释放通常通过 FIN 交换完成。

拥塞控制会通过下面这些机制动态调整发送速率：

- slow start
- congestion avoidance
- fast retransmit
- fast recovery

UDP 提供：

- 无连接传输
- 面向报文
- 不内建可靠性和顺序保证
- 协议开销低

端口号用于多路复用：

```text
(source IP, source port, destination IP, destination port)
```

这个四元组通常用来标识一个传输层会话。

## 4. Better
当应用正确性依赖可靠有序传输时，TCP 更合适，比如 Web 传输、文件传输和数据库协议。若更看重低延迟、定制可靠性，或者只是简单报文交互，UDP 往往更合适。

TCP 的优势是省心且在丢包环境下更稳妥，但代价是握手开销、重传延迟和队头阻塞。UDP 很轻，但应用必须自己处理丢包、重复、发送节奏，甚至在很多场景下自己承担拥塞适配责任。

最重要的区分之一是流量控制和拥塞控制：

- 流量控制保护接收方
- 拥塞控制保护网络

## 5. Beyond
现代传输层实践还包括：

- 基于 UDP 的 QUIC
- 选择确认
- BBR 等拥塞控制变体
- NAT 穿透问题
- 重传导致的尾延迟

更深层的结论是：传输协议不只是“给包再包一层头”。它实际上编码了端到端路径上关于可靠性、时延、公平性和故障处理的假设。
