---
title: 查找算法
---

## 1. What
查找算法的任务是判断目标是否存在于数据集合中，以及存在时它在哪里。该选什么查找方法，往往不取决于“要查什么”，而取决于“数据本身是怎样组织的”。

在数据结构课程中，最常见的对比通常是顺序查找、折半查找、分块或索引查找，以及依托树或哈希表等专门结构进行查找。

## 2. Why
查找是软件系统里最高频的基础操作之一：

- 表和索引中的定位
- 成员测试
- 已排序记录中的检索
- 结构化状态空间中的路径选择

如果查找策略选错了，再好的底层结构也可能发挥不出优势。

## 3. How
顺序查找按序扫描：

```text
for each element x:
  if x == target:
    return found
return not found
```

折半查找要求数据已排序：

```text
left = 0
right = n - 1
while left <= right:
  mid = (left + right) / 2
  if a[mid] == target:
    return found
  if a[mid] < target:
    left = mid + 1
  else:
    right = mid - 1
return not found
```

代表性的复杂度画像如下：

| Method | Requirement | Time |
| --- | --- | --- |
| linear search | 无特殊要求 | `O(n)` |
| binary search | 已排序且支持随机访问 | `O(log n)` |
| BST search | 最好是平衡搜索树 | 平均 `O(log n)` |
| hash-table lookup | 哈希分布良好 | 期望 `O(1)` |

## 4. Better
当数据量很小、未排序，或者本来就要顺序扫描时，顺序查找最直接。若数据静态、已排序，并且存放在数组这类支持随机访问的结构里，折半查找通常更优。

如果集合需要在频繁更新的同时保持有序，基于树的查找更合适；如果只关心精确匹配且不需要顺序，基于哈希的查找通常最有效。

最关键的原则是：查找复杂度由“算法 + 存储表示”共同决定。比如链表上理论上也能定义二分，但由于中间元素访问代价高，实践中几乎没有意义。

## 5. Beyond
真实系统常常会叠加多层查找机制：

- 磁盘页上的索引
- 建在树或哈希结构之上的缓存
- 在昂贵存储访问前先做 Bloom Filter 过滤
- 根据工作负载统计做查找计划优化

更深的结论是：要设计的是完整访问路径，而不是孤立地挑一个“看起来最快”的查找算法。算法再快，挂在错误的数据结构上，整体方案仍然是错的。
