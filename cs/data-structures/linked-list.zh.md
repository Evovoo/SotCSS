---
title: 链表
---

## 1. What
链表也是线性表，但元素并不放在连续内存里，而是靠指针把节点串起来。每个节点保存数据以及一个或多个指向相邻节点的链接。

常见变体包括单链表、双链表和循环链表。它们的核心特征是：结构修改主要通过改指针完成，而不是像顺序表那样整体搬移元素。

## 2. Why
链表适合下面这些情况：

- 已知位置上的插入和删除非常频繁
- 连续大块内存不容易获取
- 需要做队列链接、空闲链表或节点拼接
- 更关注节点级结构操作，而不是下标访问

它常见于图的邻接表、LRU 缓存内部结构、内存分配器以及 intrusive container 这类设计。

## 3. How
单链表节点可以抽象为：

```text
node = {
  value,
  next
}
```

头插非常直接，因为不需要整体移动元素。

```text
push_front(x):
  p = new node(x)
  p.next = head
  head = p
```

如果已知前驱节点，删除操作也是局部完成的：

```text
delete_after(prev):
  target = prev.next
  prev.next = target.next
  free(target)
```

典型操作代价：

| Operation | Time |
| --- | --- |
| access by index | `O(n)` |
| insert/delete at head | `O(1)` |
| insert/delete after known node | `O(1)` |
| search by value | `O(n)` |
| backward traversal in singly list | 单链表不直接支持 |

## 4. Better
和动态数组相比，链表避免了插入删除时的大块搬移，但代价是缓存局部性差，也无法高效按下标随机访问。

双链表通过额外的前驱指针提升了删除和逆向遍历能力，但每个节点会占用更多空间，指针维护也更复杂。循环链表则让某些循环遍历和队列结构更自然，因为尾节点能直接回到头节点。

需要特别注意的是，教材里说的 `O(1)` 插入通常默认“插入位置已经找到”。如果还要先线性寻找该位置，总成本仍然是 `O(n)`。

## 5. Beyond
链表在入门教材里常被高频使用，但在强依赖缓存性能的现代场景里并不总是优选。它最适合的，不是“看起来插删快”，而是“确实需要节点级拼接”。

典型风险包括指针错误、内存碎片、分配器开销和分支预测不友好。在托管语言里，大量小节点对象还会增加 GC 压力，所以理论上的插删优势在工程里未必真的成立。
