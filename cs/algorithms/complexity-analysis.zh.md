---
title: 复杂度分析
---

## 1. What
复杂度分析研究的是：当输入规模增长时，算法所消耗的资源如何变化。最常见的两个维度是时间复杂度和空间复杂度，通常用 `O`、`Theta`、`Omega` 等渐近记号描述。

它的目的不是精确预测某台机器上的运行秒数，而是抓住增长主趋势，便于在规模扩大时比较算法优劣。对递归算法来说，递推式和主定理是核心工具。

## 2. Why
复杂度分析之所以重要，是因为单看实现细节并不能说明可扩展性：

- `O(n log n)` 的算法在规模足够大时终究会优于 `O(n^2)`
- 内存增长有时和运行时间一样关键
- 在优化前，它能先指导算法和数据结构选型
- 它能避免把小规模 benchmark 的偶然优势误判为长期优势

它最重要的价值，是为规模增长下的决策提供理论依据。

## 3. How
标准分析流程通常是：

1. 找出基本操作
2. 统计它随输入规模 `n` 的执行次数
3. 保留主导项，忽略常数因子，进行渐近比较

对迭代算法，循环层数往往直接暴露复杂度；对递归算法，则先写递推式。

```text
T(n) = aT(n / b) + f(n)
```

这是主定理的典型形式，其中：

- `a`：子问题个数
- `b`：每个子问题的缩小比例
- `f(n)`：每一层递归之外的额外工作

主定理常见三种情形：

| Case | Condition | Result |
| --- | --- | --- |
| 1 | `f(n) = O(n^(log_b a - epsilon))` | `T(n) = Theta(n^(log_b a))` |
| 2 | `f(n) = Theta(n^(log_b a) log^k n)` | `T(n) = Theta(n^(log_b a) log^(k+1) n)` |
| 3 | `f(n) = Omega(n^(log_b a + epsilon))` 且满足 regularity | `T(n) = Theta(f(n))` |

例如归并排序：

```text
T(n) = 2T(n / 2) + O(n)
```

这里 `a = 2`、`b = 2`，而 `n^(log_b a) = n`，属于 Case 2，因此得到 `Theta(n log n)`。

## 4. Better
复杂度分析比单纯 benchmark 更适合比较增长趋势，但对真实运行时间的预测能力又弱于完整系统测量。缓存命中、分支预测、常数因子以及 I/O 成本，在中等规模下都可能主导实际表现。

对动态数组这类结构，摊还分析往往比最坏情况分析更贴切，因为少数昂贵操作会摊到大量廉价操作上。平均情况分析在输入分布可信时很有价值，但一旦分布假设不真实，就可能误导。

主定理虽然强大，但并不万能。它不适用于所有递推式，尤其是不规则划分、减法型递归，或像 `T(n-1) + T(n-2)` 这类依赖形式。

## 5. Beyond
在工程环境里，渐近复杂度还必须结合下面这些因素一起看：

- 工作负载形态
- 内存层级行为
- 并发效应
- 网络、磁盘等外部系统成本

真正要回答的问题不是孤立地问“Big-O 是多少”，而是主导增长模式是否适合当前运行环境。复杂度分析是决策框架，不是系统理解的替代品。
