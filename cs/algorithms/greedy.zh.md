---
title: 贪心算法
---

## 1. What
贪心算法按照某种局部最优规则，一步一步构造解，并且一旦做出选择通常就不再回头修改。

只有当问题满足贪心选择性质时，这种策略才是正确的，也就是：当前局部最优的决策仍然可以扩展成全局最优解。

## 2. Why
贪心方法之所以受欢迎，是因为它通常：

- 比动态规划更简单
- 更快，且更节省内存
- 实现成本低，运行流程清晰
- 很适合调度、选择和某些具有特殊结构的优化问题

一旦成立，贪心往往能用很低的复杂度解决看起来很复杂的问题。

## 3. How
设计贪心算法的一般模式是：

1. 规定一个局部排序或选择标准
2. 反复选当前最优的可行候选
3. 证明这个局部选择可以扩展为全局最优

典型例子包括：

- 按最早结束时间做区间调度
- Huffman 编码里反复合并频率最小的节点
- 最小生成树里选择安全边
- 某些特殊货币系统下的找零问题

区间调度的基本过程：

```text
sort intervals by finish time
pick first interval
for each next interval:
  if its start >= last selected finish:
    pick it
```

真正关键的部分永远是正确性证明，常见方法是交换论证或 cut argument。

## 4. Better
当问题确实允许“局部决策不可回退”时，贪心比 DP 更直接；当根本不需要穷举搜索，并且问题背后存在明确结构定理时，贪心也比回溯高效得多。

但贪心也非常容易被误用。很多问题看上去像能局部选最优，实际上却不行。比如一般形式的找零问题，或者很多路径优化问题，就必须依赖 DP 或图算法，因为单纯局部选择会导致全局错误。

“优雅而高效”和“写得很快但结果错误”之间，差别通常不在代码，而在证明。

## 5. Beyond
在实际系统里，贪心常常作为更大方案的一部分出现：

- 近似算法
- 在线算法
- 启发式调度
- 混合算法中的局部改进步骤

更深的结论是：贪心不是一种可以先猜后用的直觉，而是一种必须被证明安全的性质。如果你说不清为什么局部选择不会伤害全局最优，那通常说明它还不是一个真正成立的贪心算法。
